# AGENTS.md

## Constraints

- Build the MCP server and project tooling in TypeScript.
- Use the user's authenticated `gh` CLI and local Git credentials; do not add token storage or OAuth for v1.
- Support GitHub.com only for launch.
- Support both local repository mode and explicit `owner/repo` mode.
- Keep v1 manually invoked; do not add scheduled or CI automation unless a later plan explicitly asks for it.
- Do not embed or require a specific LLM provider in the MCP server.
- Treat commits since the last run as the guide, but allow the workflow to inspect full current repository and wiki context.
- Store wiki state visibly in `meta/state.json` and `Meta.md`; do not use hidden wiki dotfiles for durable state.
- Default to local wiki edits plus `git diff` review; never push wiki changes without explicit approval.
- Stop and ask if the wiki workspace has uncommitted changes.
- Allow creating and updating wiki pages.
- Mark stale wiki pages for review; delete or rename pages only after explicit user approval.
- Preserve Codex and Copilot as first-class harnesses. Keep harness-specific skill/instruction scaffolding under the expected structure for each tool.

## Distribution

No build/distribution playbook exists yet. If user testing requires a package build, MCP install, or harness registration before `.github/instructions/build.instructions.md` exists, ask the user how they want that build or distribution handled.

## Tech Stack

- TypeScript on Node.js 18+.
- `@modelcontextprotocol/sdk` for the local MCP server.
- `zod` for tool input and state validation.
- Vitest for unit and integration tests.
- The user's local `git` and authenticated `gh` CLI for GitHub.com repository and wiki access.

## Repo Structure

- `src/` contains the MCP server, tool handlers, Git command runner, workspace preparation, repository/wiki context, state, and wiki edit/push logic.
- `tests/` contains Vitest coverage plus Git fixture helpers and release-readiness fixtures.
- `.codex/skills/dreamers-wiki/` contains the repo-local Codex workflow scaffold.
- `.github/copilot-instructions.md` and `.github/instructions/dreamers-wiki.instructions.md` contain Copilot workflow scaffolding.
- `docs/` contains examples, troubleshooting, and release-readiness notes.
- `.dreamers-wiki/` is the ignored local workspace cache for cloned project/wiki workspaces.
- `dist/` is generated by `npm run build` and is ignored.

## Conventions

- Run `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build` before release-ready claims.
- Keep MCP tools provider-neutral: harnesses draft page content, while the server handles repository context, wiki context, local edit application, diff review, metadata, and approval-gated push.
- Require explicit `wikiPath` inputs for mutating wiki tools.
- Leave wiki edits local until the user approves the reviewed diff.
- Treat `approved:false` as a no-commit/no-push path that leaves state unadvanced.
- Mark stale pages by default; delete or rename only when the user explicitly approves the selected stale action.
- Keep durable wiki run state visible in `meta/state.json` and `Meta.md`.

## Key Files

- `src/tools.ts` registers the MCP tool surface.
- `src/workspace.ts` resolves local and explicit `owner/repo` targets and prepares clean project/wiki workspaces.
- `src/context.ts` gathers repository/wiki context and creates structured update plans.
- `src/wiki-edits.ts` applies local edits, returns review diffs, and performs approval-gated wiki pushes.
- `src/state.ts` reads and writes visible wiki state.
- `README.md` documents repo-local installation and local/explicit workflow modes.
- `docs/release-readiness.md` records final verification and smoke status.
- `tests/server-tools.test.ts`, `tests/workspace.test.ts`, `tests/context.test.ts`, `tests/wiki-edits.test.ts`, and `tests/docs-release.test.ts` cover the v1 workflow.
